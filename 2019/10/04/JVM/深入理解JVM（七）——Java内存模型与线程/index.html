深入理解JVM（七）————Java内存模型与线程实现

2019-10-04 JAVA JAVA, JVM 0 评论 字数统计: 2,850(字) 阅读时长: 10(分)

1.物理机的并发处理

现代计算机普遍都有多任务处理功能，

但是计算机的存储设备与处理器的运算速度差了好几个数量级，要如何保证处理器不会长时间处于等待资源的情况呢？于是高速缓存（Cache）应运而生了，处理器可以快速从缓存中获取资源。但是此时又引入了新的问题：缓存一致性（Cache Coherence）。在多处理器系统中，每个处理器都有自己的高速缓存，并且它们也共享同一主内存（Main Memory），当多个处理器同时操作主内存时就将产生并发问题。为了解决一致性，这些处理器在访问缓存时都会遵守一些协议，比如MSI、MESI、Dragon Protocol等以避免缓存不一致问题。

2.Java内存模型

1.主内存与工作内存

根据Java内存模型规定，所有的变量都存储在主内存中（Main Memory，类似于物理机的主内存），每条线程都有自己的工作内存（Working Memory，类似于处理器的高速缓存）。线程的工作内存中保存了被此线程使用变量的主内存副本拷贝，线程对变量的操作都必须在工作内存中完成，而不能直接读写主内存中的变量。不同线程之间也只能通过主线程完成变量传递。

2. 内存间交互

如何从主内存拷贝变量到工作内存，如何同步工作内存的变量到主内存？Java内存模型定义了以下8种原子性操作。

  • lock（锁定）：作用于主内存变量，将一个变量标识为一条线程独占状态。
  • unlock（解锁）：作用于主内存变量，将一个处于锁定变量的变量解除锁定，此时变量才可被其他线程锁定
  • read（读取）：作用于主内存变量，将一个变量的值从主内存传输到工作内存，以供load操作使用。
  • load（载入）：作用于工作内存变量，把read获取到的变量放入工作内存的变量副本中。
  • use（使用）：作用于工作内存变量，把工作内存的变量值传递给执行引擎。
  • assign（赋值）：作用于工作内存变量，把一个从执行引擎接受到的值赋给工作内存中的变量。
  • store（存储）：作用于工作内存变量，把工作内存中的变量值传输到主内存中，以供write操作使用。
  • write（写入）：作用于主内存的变量，把store操作从工作内存中获取到的值放入主内存中对于变量中。

Java内存模型规定read与load、store与write一定要保证顺序执行，当然中间可以插入其他指令，类似于：read a、read b、load b、load a.

4.Happens-Before

先行发生（Happens-Before）原则指的是Java内存模型中定义的两个操作之间的关系，比如操作A先发生于操作B，操作A产生的影响能被操作B观察到。

3.Java线程

1.线程实现

线程是比进程轻量的调度执行单位，线程可以把一个进程的分配与执行调度分开，并且各个线程又能共享进程资源。实现线程主要方式有3种：使用内核线程实现、使用用户线程实现、使用用户线程加轻量级进程混合实现。

  1. 使用内核线程实现

内核线程（Kernel-Level Thread， KLT）即直接由操作系统内核（Kernel）支持的线程，由内核来控制完成线程切换，内核通过调度器（Scheduler）对线程调度，将线程的任务映射到各处理器上。每个内核线程可以视为内核的一个分身，支持多线程的内核就叫做多线程内核（Multi-Threads Kernel）。

程序一般不会直接使用内核线程，而是使用轻量级进程（Light Weight Process， LWP），轻量级进程就是通常意义上的线程，每个LWP都由一个内核线程支持，LWP与KLT是1:1的关系。

由于内核线程的支持，每个LWP都成为一个独立调度单元，即便一个LWP在系统调用中阻塞了，也不会影响整个进程继续工作。不过LWP也有它的问题：1.由于基于内核实现，所以各种线程创建、析构、同步都需要进行系统调用，而系统调用需要在用户态（User Mode）与内核态（Kernel Mode）切换，系统耗费代价比价高。2.每个LWP都需要一个KLT支持，因此LWP也要耗费一定内核资源，因此一个系统支持的LWP是有限的。

  1. 使用用户线程实现

广义上讲，一个线程只要不是内核线程，就可以认为是用户线程（User Thread， UT），因此从LWP在此定义下实际上也是UP，但是LWP会收到系统内核限制，效率受影响。

而从狭义上定义，用户线程值完全建立在用户空间的线程库上，系统内核不能感知线程存在。用户线程的建立、同步、销毁都在用户态完成。如果程序实现得当，这种用户线程不需要切换到内核态，因此操作可以非常快速并且低消耗，也可以支持规模更大的线程数量。

使用用户线程的优势在于不需要系统内核支援，劣势也在于没有系统内核帮助，所有的线程都需用户程序处理，在一些线程问题上，比如线程阻塞问题如何处理将会非常困难。

  1. 使用用户线程与轻量级进程混合实现

在混合模式下，既存在用户线程，也存在LWP。用户线程依然在用户程序实现，所以在线程创建、切换、关闭等操作上依然开销很小，并且可以支持大规模的用户线程并发。并且操作系统内核提供的LWP作为用户线程与内核线程的桥梁，可以利用内核的线程调度功能完成许多线程问题。在此模式下，用户线程与LWP的数量比是不定的，即为N：M。

  1. Java线程模型实现

Java线程在JDK1.2前是基于用户线程实现的。而在JDK1.2中，线程模型替换为基于操作系统原生线程模型实现。对于SUN JDK来说，Windows与Linux都是使用一对一线程模型实现，即一条Java线程映射一条LWP中。

2.Java线程调度

线程调度是指系统为线程分配处理器使用权的过程，调度方式主要有两种：协同式线程调度（Cooperative Threads-Scheduling）与抢占式线程调度（Preemptive Threads-Scheduling）。

  • 使用协同式调度：

    线程的执行时间由线程本身控制，线程把自己的工作执行完后，要主动通知系统切换到另外一个线程上。协同式线程的最大好处就是实现简单，而且由于线程需要执行完才进行线程切换，切换操作是可知的，所以没有线程同步问题。缺点也很明显：线程执行时间不可控，如果某个线程编写有问题，一直没有通知系统进行线程切换，那么程序进程将会被一直阻塞。

  • 使用抢占式调度：

    每个线程将由系统分配执行时间，线程切换不由线程本身决定，在此模式下线程的执行时间是系统可控的，也不会出现某个线程导致整个进程阻塞问题。Java使用的线程调度方式就是抢占式调度。

3.Java线程状态转换

Java语言定义了5种线程状态，在任意时间点，一个线程只能有其中一种状态，这5种状态分别为：

  • 新建（New）：创建后还未启动的线程
  • 运行（Runable）：Runable包括OS线程状态中的Running和Ready，即处于此状态的线程有可能正在运行，也有可能在等待CPU分配执行时间。
  • 无限期等待（Waiting）：等待状态下的线程不会被CPU分配运行时间，它们需要等待到被其他线程显示唤醒。进入无限期等待状态的方式：
    • 未设置Timeout的Object.wait()
    • 未设置Timeout的Thread.join()
    • LockSupport.part()
  • 限期（Timed Waiting）：限期等待状态下的线程也不会被CPU分配运行时间，但是在一定时间之后它们会由系统自动唤醒。进入限期等待的方式：
    • Thread.sleep()
    • 设置了Timeout的Object.wait()
    • 设置了Timeout的Thread.join()
    • LockSupport.parkNanos()
    • LockSupport.parkUntil()
  • 阻塞（Blocked）：线程被阻塞，“阻塞状态”与“等待状态”的区别是：“阻塞状态”在等待着获取一个排他锁，这个事件将在另外一个线程放弃锁时发生；而“等待状态”则是在等待一段时间或者唤醒动作的发生。
  • 结束（Terminated）：已终止的线程状态。
